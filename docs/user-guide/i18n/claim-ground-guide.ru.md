> Эпистемическая дисциплина за 3 минуты — привязать каждое утверждение о "текущем моменте" к доказательствам времени выполнения (v1.1: also anchors standards-body term definitions to authoritative evidence; adds /claim-ground verify manual mode)

## Как работает

Claim Ground — **автоматически срабатывающая** skill. Нет slash-команды — skill активируется по характеру вопроса. Намеренный дизайн: фактический дрейф может произойти где угодно в разговоре, а ручную команду легко забыть именно тогда, когда она больше всего нужна.

| Условие срабатывания | Пример |
|---------------------|--------|
| Фактический вопрос о текущем состоянии | "Какая модель работает?" / "Какая версия установлена?" / "Что в моём PATH?" |
| Пользователь оспаривает предыдущее утверждение | "Серьёзно?" / "Уверены?" / "Я думал уже обновлено" |
| Самопроверка перед утверждением | Перед тем как Claude напишет "текущий X это Y" |

## Ключевые правила

1. **Runtime > Training** — Системный prompt, env vars и вывод инструментов всегда превосходят память. При конфликте побеждает runtime; указать источник.
2. **Сначала цитировать, потом заключать** — Вставить фрагмент доказательства ("системный prompt говорит: ...") *перед* выводом заключения.
3. **Примеры ≠ исчерпывающие списки** — `--model <model>` в помощи CLI — пример, а не полное перечисление.
4. **Оспорено → перепроверить, не перефразировать** — Когда пользователь возражает, перечитать контекст / перезапустить инструменты. Перефразирование того же неверного ответа = нарушение красной линии.
5. **Неуверен → сказать "неуверен"** — Если ни контекст, ни инструменты не могут проверить, сказать "не уверен" вместо того, чтобы гадать.

## Красные линии (нерушимые)

Красные линии — это *всегда действующие* запреты. Нарушение любой из них означает, что skill провалилась, независимо от того, как выглядит остальная часть ответа.

| # | Красная линия | Блокируемый режим галлюцинаций |
|---|---------------|-------------------------------|
| 1 | **Утверждение без источника** — делать вывод о текущем состоянии без цитирования runtime-доказательства | Factuality × extrinsic hallucination |
| 2 | **Пример как перечисление** — выводить полный список из примера CLI help / фрагмента документации | Overgeneralization |
| 3 | **Переформулировка вместо перепроверки** — повторять тот же ответ после pushback пользователя без новых вызовов инструментов (включая 3a: pushback с цитатой имеет более высокий риск) | Pushback sycophancy |
| 4 | **Утверждение о коде/API без Read/Grep** — утверждать, что символ существует / имеет сигнатуру X, не прочитав сначала определяющий файл | Code/API hallucination ([FSE 2025](https://conf.researchr.org/details/fse-2025/fse-2025-industry-papers/41/)) |
| 5 | **Цитирование без WebFetch** — цитировать URL / статью / DOI / API endpoint без реального fetch | Citation fabrication (базовая частота 19,9% по [EurekAlert 2025](https://www.eurekalert.org/news-releases/1106130)) |
| 6 | **Резюме без привязки к строкам** — резюмировать конкретный файл/PR/лог без цитирования номеров строк / разделов на каждое утверждение | Faithfulness / extrinsic addition ([HalluLens ACL 2025](https://arxiv.org/html/2504.17550v1)) |

Полные определения с сигналами, хорошими/плохими примерами и граничными случаями: [references/red-lines.md](../../skills/claim-ground/references/red-lines.md).

## Хорошие / плохие примеры

### Пример 1: текущая модель

**❌ Плохо**

> Текущая модель — Claude Opus 4.6. Это последняя модель Claude.

Чистая память. Без цитирования.

**✅ Хорошо**

> Системный prompt говорит: *"You are powered by the model named Opus 4.7 (1M context)"*. Текущая модель — Opus 4.7.

### Пример 2: список поддержки CLI

**❌ Плохо**

> Помощь CLI показывает пример `claude-sonnet-4-6`, значит CLI поддерживает только серию 4.6.

**✅ Хорошо**

> Помощь CLI показывает пример `claude-sonnet-4-6` — это placeholder, а не полный список. Полный список моделей в официальной документации.

### Пример 3: возражение пользователя

**❌ Плохо**

> **Пользователь**: Действительно 4.6? Я думал уже обновлено.
> **Agent**: Да, как я сказал, последняя серия 4.6.

**✅ Хорошо**

> **Пользователь**: Действительно 4.6? Я думал уже обновлено.
> **Agent**: Позвольте перечитать системный prompt. *"Opus 4.7 (1M context)"*. Вы правы — мой предыдущий ответ был неверным. Текущая модель — Opus 4.7.

## Playbook верификации

| Тип вопроса | Основное доказательство |
|-------------|------------------------|
| Текущая модель | Поле model в системном prompt |
| Версия CLI / поддерживаемые модели | `<cli> --version` / `<cli> --help` + официальные доки |
| Установленные пакеты | `npm ls -g`, `pip show`, `brew list` |
| Env vars | `env`, `printenv`, `echo $VAR` |
| Существование файлов | `ls`, `test -e`, инструмент Read |
| Состояние Git | `git branch --show-current`, `git log` |
| Текущая дата | Поле `currentDate` системного prompt или команда `date` |

## Взаимодействие с другими forge skills

### С block-break

**Ортогональные, взаимодополняющие**. block-break говорит "не сдавайся"; claim-ground говорит "не утверждай без доказательств".

Когда оба срабатывают: block-break предотвращает капитуляцию, claim-ground принуждает к перепроверке.

### С skill-lint

**Разные категории**. skill-lint — это **anvil** (валидирует статические файлы плагина, выдаёт pass/fail); claim-ground — это **hammer** (ограничивает собственный эпистемический вывод Claude во время выполнения). Обязанности не пересекаются.

## FAQ

### Почему нет slash-команды?

Фактический дрейф может произойти в любом ответе. Ручную команду легко забыть именно в те моменты, когда она нужна. Автоматическое срабатывание по описанию надёжнее.

### Срабатывает ли на каждый вопрос?

Нет. Только на две конкретные формы: **текущее/живое состояние системы** или **возражение пользователя на предыдущее утверждение**.

### А если я хочу, чтобы Claude угадывал?

Переформулировать как "сделай обоснованное предположение о X" или "вспомни из обучающих данных: X". Claim Ground поймёт, что это не вопрос о runtime.

### Как узнать, что сработало?

Искать паттерны цитирования в ответе: `системный prompt говорит: "..."`, `вывод команды: ...`. Доказательство перед заключением = сработало.

## Когда использовать / Когда НЕ использовать

### ✅ Используйте когда

- Asking about current system state (model version, installed tools, env, config)
- Challenging a prior factual assertion ("really? / are you sure?")
- Before Claude is about to claim something about "right now"

### ❌ Не используйте когда

- Pure reasoning tasks (algorithms, math, type inference)
- Creative / brainstorming work
- Explaining training-knowledge concepts (e.g., "what is async/await")

> Шлюз для фактических утверждений — гарантирует цитаты, не их правильность, не обрабатывает не-фактическое.
